### Summary

BIP32 HD wallets allow a tree of key pairs to be derived from a single master seed. Their defining feature is **non-hardened derivation**: child public keys can be derived from a parent public key alone, enabling watch-only wallets where a server generates fresh receiving addresses while the signing key stays offline. Classical BIP32 achieves this via the linear algebraic structure of elliptic curve public keys (child pubkey = parent pubkey + scalar·G).

This paper is the first to recover that full functionality in the post-quantum setting with provable security under standard lattice assumptions (M-LWE, M-SIS).

### Why Lattices Break Naive BIP32

Two structural obstacles prevent straightforward translation:

1. **Rounding:** ML-DSA (standardized Dilithium) drops low-order bits from the public key (`t̂ = ⌊As + e⌉_νt`) to reduce key size. Rounding is a lossy, non-linear operation — it does not distribute over addition, making non-hardened derivation structurally impossible for any scheme that rounds its public keys.

2. **Noise accumulation:** Even without rounding, lattice public keys have the form `t = As + e`. Each derivation step adds fresh noise, widening the Gaussian distribution and potentially making derived keys distinguishable from freshly generated ones, breaking unlinkability.

### Construction 1 — ML-DSA HD Wallet (hardened derivation only)

Uses NIST-standardized ML-DSA. Child keys are generated by expanding an HMAC output as a deterministic seed into a fresh ML-DSA key pair via `DetKeyGen`. Because ML-DSA rounds its public key, `CKDerpub` cannot be defined — **all derivation requires the private key**.

```
Setup:    sd, cc ← {0,1}^λ;  (sk, pk) := SIG.DetKeyGen(seed)
CKDer:    (sd', cc') := HMAC(xpriv, cc, t)
          (sk', pk') := SIG.DetKeyGen(sd')
```

**Security:** Unlinkable and unforgeable under the random oracle model for HMAC. Unlinkability advantage is bounded by `q_hmac·(q_pk+1)/2^λ + ε_ML-DSA`. Comparable in scope to QIP2 [Smi] but includes a formal security analysis that QIP2 lacks.

### Construction 2 — Raccoon-G HD Wallet (non-hardened derivation, primary contribution)

Uses **Raccoon-G**, a variant of the Raccoon signature scheme (introduced in Threshold Raccoon [DKM+24]) that replaces sums-of-uniform secrets with **discrete Gaussian secrets**. Key modifications:

- The full, unrounded public key `t = As + e` is published (not `t̂`). Rounding to `t̂` is performed inline only during `Sign` and `Verify`, preserving linearity for derivation.
- `RandPK(t, ρ)`: samples fresh `(s', e') ← D^ℓ_σt × D^k_σt` via a deterministic `Sam(ρ)` expander, then returns `t' = As' + e' + t`.
- `RandSK(s, ρ)`: returns `s + s'`.

Non-hardened child derivation:
```
CKDerpub:   (ω, cc') := HMAC(pk, cc, t)
            pk' := SIG.RandPK(pk, ω)

CKDerpriv:  (ω, cc') := HMAC(pk, cc, t)
            sk' := SIG.RandSK(sk, ω);  pk' := SIG.RandPK(pk, ω)
```

**Why Gaussians work:** By Lemma 2 (Micciancio–Peikert), the sum of independent Gaussians of width σ is statistically close to a Gaussian of width σ√n at depth n. This means derived keys remain in the same distributional family as freshly generated ones, enabling provable unlinkability.

**Signature correctness under rerandomization:** The verification bound widens with derivation depth:

```
B_{2,z} = e^{1/4} · (√(n+1)·τ·σ_t + σ_w)·√(N(k+ℓ)) + (τ·2^νt + 2^(νw+1))·√(Nk)
```

**Security:** Unlinkable and unforgeable under M-LWE, M-SIS, and H-MLWE (Hint-MLWE) assumptions. The unlinkability proof uses a five-hybrid argument showing that depth-n signatures are indistinguishable from depth-1 signatures, relying on H-MLWE hardness at both depth 1 and depth n. Unforgeability is proved via UF-CMA-HRK (unforgeability under honestly rerandomized keys).

**Bounded derivation depth:** Unlike classical BIP32 (where non-hardened derivation depth is unlimited), noise accumulation imposes a maximum tree depth `n` fixed at key generation time. In practice this matches BIP44's deployment pattern: hardened derivation for the first three levels (purpose, coin type, account) resets noise, and non-hardened derivation at the final two levels (change, address index) branches from those hardened roots.

**Chain code security:** Identical caveat to classical BIP32 — if an attacker obtains a child's secret key, they can recompute the derivation offset and recover the parent secret key for all siblings in the non-hardened subtree. Hardened nodes eliminate this by deriving the offset from the parent private key.

### Scheme Comparison

| Feature                    | ML-DSA               | Raccoon-G                          |
|----------------------------|----------------------|------------------------------------|
| Secret distribution        | Bounded uniform      | Discrete Gaussian                  |
| Public key structure       | Rounded (lossy)      | Full (linear)                      |
| Non-hardened derivation    | Not possible         | Supported via Gaussian stability   |
| Signing architecture       | Rejection sampling   | Fiat-Shamir without aborts         |
| NIST status                | Standardized (FIPS 204) | Not on NIST track               |
| Hardware implementation    | Straightforward      | Challenging (Gaussian sampling)    |

### Status in This Codebase

`src/crypto/dilithium_hd_key.{h,cpp}` provides the BIP32 scaffolding (depth, fingerprint, chaincode, hardened/non-hardened dispatch) but the actual key derivation is a placeholder: the non-hardened paths simply XOR 32 HMAC bytes into the first 32 bytes of a 1312-byte public key, which is not a valid or secure operation. Neither Construction 1 nor Construction 2 from this paper has been implemented. In particular:

- Construction 1 requires a proper `DetKeyGen` that derives a full ML-DSA key pair from a fixed seed.
- Construction 2 requires implementing Raccoon-G (not present in this codebase at all), including the `RandPK`/`RandSK` algorithms and publishing unrounded public keys.
